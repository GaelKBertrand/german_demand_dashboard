# GATI Skills-Mobility Dashboards: Germany 2026

A static, GitHub-Pages-ready site: a landing page that links to one dashboard
per sector. Each dashboard reads that sector's **clean CSV** at runtime and
builds the charts in the browser — no server, no build step.

## >>> Where the CSV is loaded <<<

Two files, one job:

1. **`assets/js/sectors.js`** defines each sector and its CSV path, and the
   `CSV_BASE` that all paths resolve against. This is the single source of truth
   for *where* data comes from.
2. **`assets/js/data-loader.js`** does the actual `fetch()` (inside
   `loadSectorData()`), parses the CSV with PapaParse, and builds the data model.

**Default (recommended):** put the clean CSVs in `data/` and leave
`CSV_BASE = ""`. The dashboard fetches `data/healthcare.csv`, etc. — relative to
the site, so it works on GitHub Pages project sites without worrying about the
`/<repo>/` prefix.

**Alternative:** if the CSVs live in another repo, set
`CSV_BASE = "https://raw.githubusercontent.com/<user>/<repo>/main/"` in
`sectors.js` and keep each `csv: "data/<sector>.csv"`.

## Expected CSV columns

The clean output of the sector classifiers. The loader is tolerant of extras and
missing columns, but uses these when present:

`State`, `Employer_Category`, `ISCO_2`, `ISCO_3`, `ISCO_4`, `ISCO_4_name`,
`Employment_type`, `Date_Posted`, `Company_Name`, `Job_Title`, `Job_URL`,
`Scope_Category` (rows marked `Out of Scope` / `CLASSIFICATION_FAILED` are
excluded from the analysis).

State names should be the English Bundesland names the classifiers emit
(e.g. `Bavaria`, `North Rhine-Westphalia`) — the regional map keys on those.

## Live counts on the landing page

The landing page shows each sector's posting count and date range without ever
downloading a full CSV. Those numbers come from tiny `data/<sector>.meta.json`
files (a few hundred bytes each), generated by `scripts/build-meta.mjs`:

```
npm run build:meta        # or: node scripts/build-meta.mjs
```

The script scans every `data/*.csv`, counts the classified rows, and writes
`data/<sector>.meta.json` (total, #states, date range, top occupation, build
date). If a sector's meta file is missing, its card just omits the count line —
nothing breaks. You can run it locally, but you normally don't have to: the
GitHub Action below runs it for you on every push.

## Publish on GitHub Pages (via Actions)

This repo deploys through GitHub Actions, so `meta.json` is always regenerated
from the current CSVs at deploy time.

1. Push this folder's contents to the **root** of a repo, on `main`.
2. Repo **Settings → Pages → Build and deployment → Source: GitHub Actions**
   (not "Deploy from a branch").
3. That's it. On every push to `main`, `.github/workflows/deploy.yml`:
   rebuilds the `meta.json` files → uploads the site → deploys to Pages.
   The live URL appears in the Action's summary and under Settings → Pages.

To update the data: replace/add a `data/<sector>.csv`, commit, push. The Action
refreshes the counts and redeploys. No server, ever.

### Local preview

Serve over http:// (the CSV `fetch` needs a real origin — opening the file
directly hits CORS):

```
npm run build:meta        # generate meta.json for local counts
python3 -m http.server 8000
# open http://localhost:8000/
```

## Add a new sector

1. Drop `data/<sector>.csv` into `data/`.
2. Add one entry to the `SECTORS` array in `assets/js/sectors.js`
   (id, label, csv path, tagline, scope, accent colour).
   A card appears on the landing page automatically, linking to
   `dashboard.html?sector=<id>`.

## Design / structure notes

- The dashboard keeps the original data model unchanged: `APP` (filter state),
  `DATA` (`lookup` + 8-integer `rows` + `roleTable` / `stateGeo` / `companies` /
  `records`), and `getRows()`. Only the *source* changed — `data-loader.js`
  reconstructs that `DATA` shape from the CSV instead of it being pre-baked into
  the HTML — and the visuals were refreshed to the GATI identity.
- Libraries load from cdnjs: PapaParse, Plotly, Leaflet. Fonts from Google Fonts.
